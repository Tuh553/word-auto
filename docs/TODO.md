## 已完成基线

- 工程安全网：`pnpm run ci` 串联 `typecheck`、`test`、`build`；GitHub Actions
  `quality.yml` 覆盖 PR / push。
- 金标准回归：基于 `templates/source/*.docx` 固化 parser 解析、validator 分类与校验基线。
- parser 结构信号：已识别 `w:drawing`、OMML、`w:object` 的存在性与所在段落定位，
  并接入图题注 / 公式编号行分类降噪。
- Web/Parser 错误分流：已区分非 zip、加密 docx、损坏 zip、旧版 `.doc`、非 OOXML 结构，
  并在前端映射为明确中文提示。
- 规则闭环：字段值编辑、`exact` / `oneOf` / `range` / `unset`、草稿保存、发布、发布后回灌检测。
- 多模板 MVP：内置模板、自定义规则库 JSON 导入/导出、模板切换。
- 模板候选 MVP：上传 `.docx` -> parser + classify -> 候选统计 -> 接受到草稿 -> 发布 -> 检测消费新规则。

## P0：保持安全网有效

1. 任何 parser / classify / validate 行为变化，都必须同步更新或新增金标准测试。
2. 每次改规则模型或候选提取，都要跑 `pnpm run ci`；不要只跑 web build。
3. 文档状态变更后同步更新 `README.md`、`PROGRESS.md` 和本文件，避免历史路线图反向误导。

## P1：解析准确率

1. 页眉/页脚结构化解析：用 XML 解析替代正则文本抓取，区分左/中/右与页码域。
2. `numbering.xml`：解析自动编号，支撑标题题序连续、图表题注连号、列表识别。
3. 域 / 题注 / 交叉引用 / 脚注尾注：至少识别存在、文本与所在段落。
4. 表格全局顺序：当前已提取表格内段落，但未保留表格与正文交错顺序；如改 `preserveOrder`
   必须先扩充金标准。
5. run 级混排：校验段落内局部 run 字体/字号不一致，定位到 run 或字符区间。
6. 附录细分：拆分附录内部小标题、成果清单、落款等角色。

## P1：校验与报告可信度

1. 统计型文档检测：参考文献条数/外文占比、关键词数量、摘要字数区间。
2. 规则依据展示：把规则库 `source.provenance` 接入报告，显示原始规范依据。
3. 修复建议：为 issue 增加可操作的人话指引，区分可自动修复 / 需手动处理。
4. 角色识别置信度：对启发式命中的低置信段落打标，减少“看似确定”的误导。
5. 报告分组与排序：按角色、严重级、字段、章节聚合。

## P1：规则库与模板候选

1. 多模板管理补齐：在 UI 里支持新建、复制、重命名、删除模板；目前主要靠导入/切换。
2. 候选 diff：展示候选与当前草稿的差异，支持忽略候选并记录状态。
3. 多样本聚合：支持上传多篇样本文档共同提取候选，降低单文档偶然性。
4. 候选证据下钻：展示样本段落、角色来源、冲突值明细。
5. 候选评分校准：当前是 MVP 置信提示；后续要把样本数、覆盖率、冲突率、角色可靠性统一成更稳定的评分。

## P2：Web 体验与交付

1. Web Worker：解析与校验移出主线程，避免大文档阻塞 UI。
3. 预览联动：默认定位到首个问题，报告与预览双向滚动，高亮支持 run 级区间。
4. 批量检测：一次上传多篇 `.docx`，输出汇总与单篇下钻。
5. 带批注 docx 导出：纯 OOXML 生成批注，不改写正文。

## P3：平台化与远期

1. 后端持久化 / API：把当前 `localStorage` 草稿和规则库升级为可选服务端存储。
2. 规范生态：多校/多单位/多语言规则库，支持规则库贡献与版本管理。
3. Word/WPS 插件：在编辑器内实时检测。
4. 自动套版 / 一键修复：远期高风险，只能走纯 OOXML；必须无损保留分节、域、题注、交叉引用、编号，绝不引入 Word COM。

## 已知边界

- 模板候选只基于已识别段落；未识别段落不会进入统计。
- 行距候选只统计显式固定/最小行距；多倍行距不会自动折算为 pt。
- 当前前端规则库持久化是浏览器 `localStorage`，不是多人协作或服务端审批流。
- 表格段落已提取，但与正文的全局交错顺序尚未保留。
- OMML / 嵌入对象目前只识别存在性和所在段落，不展开公式内容语义。
