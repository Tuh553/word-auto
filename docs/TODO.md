## P0：保持安全网有效

1. 任何 parser / classify / validate 行为变化，都必须同步更新或新增金标准测试。
2. 每次改规则模型、候选提取或 Web 状态流，都要跑 `pnpm run ci`；不要只跑 web build。
3. 文档状态变更后同步更新 `README.md`、`PROGRESS.md` 和本文件，避免历史路线图反向误导。

## P1：解析准确率、校验可信度、规则库与模板候选

**详细开发计划见 [`docs/P1-PLAN.md`](./P1-PLAN.md)，总工作量 5-8 周。**

### P1 核心目标

1. **多模板管理补齐**：UI 支持新建、复制、重命名、删除模板
2. **候选 diff 与证据下钻**：展示候选与草稿差异，支持忽略，展示样本段落、角色来源、冲突值明细
3. **多样本聚合与评分校准**：上传多篇样本共同提取候选，统一评分模型
4. **表格全局顺序**：保留表格与正文交错顺序
5. **附录细分**：拆分附录内部小标题、成果清单、落款等角色

## P2：Web 体验与交付

1. Web Worker：解析与校验移出主线程，避免大文档阻塞 UI。
2. 预览联动：报告与预览双向滚动，高亮支持 run 级区间。
3. 批量检测：一次上传多篇 `.docx`，输出汇总与单篇下钻。
4. 带批注 docx 导出：纯 OOXML 生成批注，不改写正文。

## P3：平台化与远期

1. 后端持久化 / API：把当前 `localStorage` 草稿和规则库升级为可选服务端存储。
2. 规范生态：多校/多单位/多语言规则库，支持规则库贡献与版本管理。
3. Word/WPS 插件：在编辑器内实时检测。
4. 自动套版 / 一键修复：远期高风险，只能走纯 OOXML；必须无损保留分节、域、题注、交叉引用、编号，绝不引入 Word COM。

## 已知边界

- 模板候选只基于已识别段落；未识别段落不会进入统计。
- 行距候选只统计显式固定/最小行距；多倍行距不会自动折算为 pt。
- 当前前端规则库持久化是浏览器 `localStorage`，不是多人协作或服务端审批流。
- 表格段落已提取，但与正文的全局交错顺序尚未保留。
- OMML / 嵌入对象目前只识别存在性和所在段落，不展开公式内容语义。
- 自动编号、段落域和题注 / 交叉引用关联已解析；交叉引用有效性已接入 `validateDoc`，后续新增域类型时需同步补回归。
- 页眉/页脚已识别基础文本、位置、`PAGE` 域和常用样式；后续如扩展更多边框/页码格式规则需同步补回归。
