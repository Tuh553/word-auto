# 实施计划：多模板规则库可视化配置与模板候选提取

> 状态更新（2026-06-11）：阶段 1-7 的 MVP 已落地并通过 `pnpm run ci`。
> 2026-06-15：Web 工作台已完成结构拆分，`App` 只做导航和编排，检测、规则库、候选提取状态分别进入 hooks。
> 当前 TODO 集中维护在 [`../TODO.md`](../TODO.md)；本文保留实施过程与历史决策。

## 1. 目标

将 [`2026-06-09-rule-library-and-template-prd.md`](./2026-06-09-rule-library-and-template-prd.md)
拆解为可执行的工程任务，优先完成第一阶段：

- 规则库数据模型重构
- 规则配置页
- 规则合法性校验

第二阶段模板候选提取在第一阶段稳定后推进。

## 2. 实施原则

- 先统一规则模型，再做 UI
- 先做草稿态与生效态分离，再做模板候选接入
- validator 只读取正式规则，避免半编辑状态污染检测链路
- 第一阶段不引入真实文档重跑，不把范围扩成完整分析工作台

## 3. 阶段拆分

### 阶段 1：规则模型与存储层

目标：

- 定义 `RuleLibrary`、`RuleDraft`、`RoleRuleSet`、`RuleField`、`RuleValue`
- 为现有规则库 JSON 提供迁移/兼容层
- 为后续前端配置页与后端校验提供统一接口

任务：

1. 梳理当前规则库 JSON 结构与 validator 消费路径
2. 定义新的 TypeScript 类型与 schema
3. 实现旧规则到新结构的转换器
4. 实现新结构到 validator 读取格式的适配层
5. 增加基础单元测试，覆盖：
   - `exact`
   - `oneOf`
   - `range`
   - `unset`
   - 禁用字段

交付物：

- 新规则类型定义
- 规则迁移/适配模块
- 覆盖核心字段模式的测试

验收：

- 现有 CLI 校验链路可继续跑通
- 不需要修改 validator 主逻辑即可读取新规则结构，或仅需有限适配

### 阶段 2：草稿态与发布态

目标：

- 在规则库层面明确“草稿”和“生效”分离
- 为前端编辑流程提供稳定状态模型

任务：

1. 设计规则存储目录与文件命名约定
2. 定义草稿保存格式
3. 定义发布动作：
   - 草稿合法性校验通过
   - 生成新生效版本
   - 更新当前规则索引
4. 定义最小版本元数据：
   - `id`
   - `name`
   - `version`
   - `updatedAt`
   - `status`

交付物：

- 草稿/发布状态流转规则
- 最小存储规范

验收：

- 用户编辑不会直接覆盖生效规则
- 发布动作可明确区分于保存草稿

### 阶段 3：规则合法性校验

目标：

- 提供规则编辑期间的结构性和业务提示校验

任务：

1. 实现结构校验：
   - 必填角色
   - 必填字段
   - 数值/枚举合法性
2. 实现一致性校验：
   - `min <= max`
   - `oneOf` 非空
   - 字段单位匹配
3. 实现业务提示：
   - 标题与正文层级区分不足
   - 关键角色缺失
   - 字体配置不完整
4. 定义返回结果结构：
   - `errors`
   - `warnings`
   - `infos`

交付物：

- 规则合法性校验模块
- 校验结果类型定义
- 单元测试

验收：

- 前端可拿到字段级和页面级校验信息
- 校验结果不依赖真实 `.docx` 输入

### 阶段 4：规则配置页

目标：

- 提供业务用户可用的 web 配置页

任务：

1. 页面骨架：
   - 左侧角色列表
   - 右侧字段编辑区
2. 字段渲染组件：
   - 开关
   - 模式选择
   - 数值输入
   - 枚举选择
   - 严重级别选择
3. 页面状态：
   - 脏状态
   - 保存中
   - 发布中
   - 校验中
4. 页面操作：
   - 保存草稿
   - 执行合法性校验
   - 发布规则
5. 结果反馈：
   - 字段级错误提示
   - 页面级校验汇总
   - 当前状态标签

交付物：

- 规则配置页
- 基础交互组件
- 与后端接口联调

验收：

- 用户无需编辑 JSON 即可完成常见规则修改
- 能看到未保存状态和校验结果

### 阶段 5：多规则库管理基础能力

目标：

- 支持多个模板规则库共存

任务：

1. 规则库列表页或选择器
2. 新建规则库基础流程
3. 基于已有规则复制一份新规则库
4. 规则库元数据展示

交付物：

- 规则库列表/切换能力

验收：

- 至少支持查看和切换多个规则库

### 阶段 6：模板候选提取

目标：

- 从 `.docx` 模板或样本文档中提取候选规则

任务：

1. 定义 `RuleProposal` 与 `RuleProposalField`
2. 基于现有 parser + classify 做角色聚合
3. 为每个角色字段统计：
   - 主值
   - 样本数
   - 覆盖率
   - 冲突值
4. 初版置信度算法
5. 候选结果接口输出

交付物：

- 模板候选提取模块
- 候选接口
- 覆盖典型模板/样本文档的测试样例

验收：

- 系统可输出每个角色的候选字段结果
- 候选结果包含证据与冲突信息

### 阶段 7：候选导入草稿

目标：

- 允许用户把模板候选逐项接受到草稿

任务：

1. 候选面板 UI
2. 当前草稿与候选的 diff 展示
3. 单字段接受
4. 单角色批量接受
5. 接受后重新跑规则合法性校验

交付物：

- 候选导入交互

验收：

- 候选项不会直接覆盖正式规则
- 接受候选后会进入草稿态

## 4. 推荐任务顺序

推荐按以下顺序开工：

1. 规则类型定义与兼容层
2. 规则合法性校验模块
3. 草稿/发布态存储设计
4. 规则配置页骨架
5. 字段编辑组件
6. 保存/校验/发布接口
7. 多规则库列表与切换
8. 模板候选提取
9. 候选导入草稿

原因：

- 规则模型是后续所有工作的共用基础
- 合法性校验越早落地，前端越不容易变成“盲编辑”
- 候选提取必须建立在稳定规则结构之上

## 7. 风险控制

### 风险 1：旧规则迁移成本高

应对：

- 先写兼容层，再逐步替换内部消费方

### 风险 2：前端先做，后端模型反复改

应对：

- 强制先冻结规则字段结构，再做页面

### 风险 3：候选提取误导用户

应对：

- 候选只进草稿
- 必须展示冲突与置信度

### 风险 4：一期范围膨胀

应对：

- 不接入真实文档重跑
- 不做复杂版本中心
- 不做条件表达式规则编辑
