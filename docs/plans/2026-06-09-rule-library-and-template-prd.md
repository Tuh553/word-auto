# PRD：多模板规则库可视化配置与模板候选提取

## 1. 背景

当前项目已经具备 `.docx` 解析、段落角色识别、规则校验与报告输出能力，但规则库仍偏向内部使用：

- 规则主要以 JSON 维护，缺少业务用户可操作的配置界面
- 仅支持有限的内置模板，无法支撑多学校/多单位/多规范场景
- 缺少从标准模板或样本文档中提取规则候选的能力

这导致产品在“扩模板”和“交付给非开发用户配置规则”两个方向上存在明显阻塞。

本 PRD 定义一个两阶段方案：

1. 第一阶段：提供业务用户可用的规则可视化配置页与规则合法性校验
2. 第二阶段：支持上传 `.docx` 模板或样本文档，自动提取规则候选，供用户确认并写入规则草稿

## 2. 目标

### 2.1 产品目标

- 让业务用户无需直接编辑 JSON，即可查看、修改、保存和发布规则库
- 让系统支持多个模板规则库，而非单一内置规则
- 让用户可以基于标准模板或样本文档，快速生成规则候选并人工确认

### 2.2 工程目标

- 建立稳定、可扩展的规则数据模型，供前端配置页、后端校验器和模板候选提取共用
- 将“正式规则”“编辑草稿”“模板候选”三类对象分层，避免状态混淆
- 为后续 web 工作台打下稳定接口，而不是继续依赖松散 JSON

## 3. 非目标

以下内容不在本次范围内：

- 不实现 Word COM / Office 自动化
- 不做自动改写文档或自动套版
- 第一阶段不做真实文档重跑与结果预览
- 第一阶段不做复杂版本管理、审批流或多人协作
- 第一阶段不开放复杂条件表达式规则，例如“仅表格内生效”“仅某章节生效”

## 4. 用户与场景

### 4.1 目标用户

- 维护学校/单位规范的业务人员
- 项目内部负责规则接入和模板扩展的实施人员

### 4.2 典型场景

1. 业务人员打开规则配置页，查看“正文”“一级标题”“摘要正文”等角色的格式规则，并修改字号、字体、对齐方式
2. 用户保存草稿并进行规则合法性校验，确认无结构性问题后发布为生效规则
3. 用户上传学校标准模板 `.docx`，系统自动提取正文、标题等角色的候选规则，展示置信度和冲突项，供用户确认导入草稿
4. 用户上传一篇“看起来合规”的样本文档，系统同样尝试归纳候选规则，但以更保守方式呈现结果

## 5. 总体方案

### 5.1 分阶段策略

#### 第一阶段：规则库可视化配置

提供业务用户可用的规则配置页，支持：

- 查看已有规则库
- 编辑各文档角色下的规则字段
- 保存草稿
- 发布生效规则
- 执行规则合法性校验

#### 第二阶段：模板候选提取

支持上传 `.docx` 标准模板或样本文档，系统基于现有 parser 与 validator/classify 能力提取候选规则，展示：

- 建议值
- 覆盖率
- 样本数
- 冲突值
- 置信度

用户可选择将候选项接受到规则草稿，而不是直接覆盖正式规则。

### 5.2 核心原则

- `RuleLibrary` 是唯一正式生效规则源
- `RuleDraft` 承接人工编辑中的未发布状态
- `RuleProposal` 承接模板解析得到的候选规则
- validator 仅消费正式规则，不读取草稿或候选
- 所有前端交互使用统一规则字段结构，不直接暴露底层 OOXML 单位

## 6. 信息架构与数据模型

### 6.1 核心对象

#### RuleLibrary

正式生效的规则库，用于实际校验。

#### RuleDraft

规则库的编辑草稿。结构与 `RuleLibrary` 基本一致，但允许出现：

- 未发布修改
- 字段级错误
- 规则合法性告警

#### RuleProposal

由标准模板或样本文档解析得到的候选规则集合，不直接参与校验。

### 6.2 规则字段模型

每个文档角色下的每个可校验属性统一为标准字段结构。

```ts
type RuleFieldKey =
  | "fontFamilyCn"
  | "fontFamilyLatin"
  | "fontSizePt"
  | "bold"
  | "align"
  | "lineHeight"
  | "firstLineIndentChars"
  | "spaceBeforePt"
  | "spaceAfterPt";

type RuleSeverity = "error" | "warn" | "info";
type RuleValueMode = "exact" | "oneOf" | "range" | "unset";

interface RuleValue {
  mode: RuleValueMode;
  exact?: string | number | boolean;
  oneOf?: Array<string | number>;
  min?: number;
  max?: number;
  unit?: "pt" | "chars" | "multiple" | "enum" | "bool";
}

interface RuleField {
  key: RuleFieldKey;
  label: string;
  enabled: boolean;
  severity: RuleSeverity;
  value: RuleValue;
  note?: string;
}
```

### 6.3 角色模型

规则字段按文档角色分组，例如：

- 封面标题
- 中文摘要标题
- 中文摘要正文
- 关键词
- 一级标题
- 二级标题
- 三级标题
- 正文
- 参考文献

建议结构：

```ts
interface RoleRuleSet {
  role: string;
  label: string;
  fields: RuleField[];
}

interface RuleLibrary {
  id: string;
  name: string;
  version: string;
  basedOn?: string;
  roles: RoleRuleSet[];
}
```

### 6.4 候选规则模型

模板解析结果以候选字段形式保存：

```ts
interface RuleProposalField {
  key: RuleFieldKey;
  proposedValue: RuleValue;
  confidence: number;
  sampleCount: number;
  coverage: number;
  evidence: string[];
  conflicts?: Array<{
    value: RuleValue;
    sampleCount: number;
  }>;
}
```

说明：

- `confidence` 表示系统对该建议值的置信度
- `sampleCount` 表示支持该建议的段落数
- `coverage` 表示该建议覆盖该角色的比例
- `evidence` 用于解释建议来源
- `conflicts` 展示竞争值，帮助用户判断是否存在异常样本或角色识别问题

## 7. 第一阶段：规则配置页 PRD

### 7.1 页面目标

提供一个业务用户可操作的规则配置页，不需要理解 JSON 即可完成规则维护。

### 7.2 页面结构

建议采用双栏布局：

- 左侧：角色列表
- 右侧：当前角色的字段编辑区

### 7.3 左侧角色列表

按文档语义分组展示，示例：

- 前置部分：封面、摘要标题、摘要正文、关键词、目录标题
- 正文部分：一级标题、二级标题、三级标题、正文
- 末尾部分：参考文献、附录标题

每个角色展示：

- 角色名称
- 已启用字段数量
- 是否存在未保存修改
- 是否存在校验错误

### 7.4 右侧字段编辑区

每个字段一行，包含：

- 字段名
- 是否启用校验
- 规则方式：精确值 / 候选值之一 / 范围 / 不校验
- 规则值
- 严重级别
- 备注

交互要求：

- 单位统一友好显示，例如 `pt`、`字符`、`倍`
- 对齐方式使用受限选项，不暴露底层 OOXML 值
- 字体字段支持常用值下拉 + 手工输入
- 字段禁用后，不参与校验
- 编辑后立即标记草稿脏状态

### 7.5 页面级操作

第一阶段至少提供：

- `保存草稿`
- `发布规则`
- `执行规则合法性校验`

不在第一阶段提供：

- 真实文档重跑分析
- 结果预览联动
- 多版本回滚 UI

### 7.6 交互反馈

需要明确显示：

- 当前编辑对象名称
- 当前是否为草稿态
- 上次保存时间
- 是否存在未通过的校验项

## 8. 第一阶段：规则合法性校验

第一阶段只做规则合法性校验，不做“用真实文档验证规则效果”的闭环。

### 8.1 结构校验

- 必填角色是否存在
- 必填字段是否为空
- 数值字段是否可解析
- 枚举值是否在允许集合中

### 8.2 一致性校验

- `range` 的 `min <= max`
- `oneOf` 至少有一个候选值
- 被禁用字段不应保留无意义冲突配置
- 字段单位与字段类型匹配

### 8.3 业务提示

提供非阻断提示，例如：

- 正文字号与一级标题字号相同，可能缺乏层级区分
- 中文字体与西文字体都未配置，可能降低字体检测稳定性
- 关键角色完全未配置，规则覆盖可能不完整

### 8.4 边界说明

规则合法性校验只能说明：

- 规则结构上是否成立
- 配置是否存在明显冲突或缺漏

不能说明：

- 该规则一定符合某学校规范
- 该规则在所有真实文档上一定效果正确

该边界需要在 UI 上明确提示。

## 9. 第二阶段：模板候选提取 PRD

### 9.1 输入来源

支持两类输入：

1. 学校/单位标准模板 `.docx`
2. 用户自行整理的样本文档

系统设计必须按第二类场景兜底，不能假设输入模板绝对规范或绝对纯净。

### 9.2 处理流程

建议流程如下：

1. 上传 `.docx`
2. 调用现有 parser 解析文档模型与有效格式
3. 调用现有 `classify` 对段落进行角色识别
4. 以角色为单位聚合字段分布
5. 为每个字段生成候选值、覆盖率、冲突项和置信度
6. 输出 `RuleProposal`
7. 在前端展示候选项，允许用户选择“接受到草稿”或忽略

### 9.3 候选项展示要求

每个候选字段至少展示：

- 字段名
- 建议值
- 置信度
- 支持样本数
- 覆盖率
- 冲突值列表
- 证据说明

示例：

- 角色：正文
- 字段：字号
- 建议：`12 pt`
- 置信度：`0.93`
- 覆盖率：`84 / 90`
- 冲突：`9 pt` 出现 `6` 次

### 9.4 接受候选的交互

候选项不能直接覆盖正式规则，只能：

- 接受为草稿字段值
- 与当前草稿对比后局部替换
- 保留原值并忽略候选

### 9.5 置信度原则

对于标准模板：

- 更看重样式定义稳定性
- 更看重高覆盖率

对于样本文档：

- 更看重聚类稳定性
- 对离群值更保守
- 对低覆盖角色降低置信度

第一阶段无需实现该算法；第二阶段落地时再细化评分规则。

## 10. 功能边界与约束

### 10.1 规则编辑器边界

第一阶段不暴露复杂条件规则，例如：

- 仅表格内生效
- 仅某章节生效
- 中文与西文字体的条件联动表达式

原因：

- 这类逻辑应优先保留在 validator 内建能力中
- 若提前暴露，将把配置页演变为低配规则编程语言，显著增加维护成本

### 10.2 技术约束

- 继续坚持纯 OOXML 路线，不引入 Word COM
- 规则展示层不得暴露 `twips`、half-point 等底层内部单位
- validator 仅读取正式规则库，不直接消费模板候选或脏草稿

## 11. 验收标准

### 11.1 第一阶段验收

- 用户可在 web 端查看规则库角色与字段
- 用户可修改常见字段，如字体、字号、加粗、对齐、行距、首行缩进
- 用户可保存草稿并发布为生效规则
- 系统可执行规则合法性校验，并给出错误与提示
- 不需要直接编辑 JSON 即可完成基础规则维护

### 11.2 第二阶段验收

- 用户可上传标准模板或样本文档
- 系统可输出分角色的候选规则
- 每个候选项包含建议值、置信度、覆盖率、冲突信息
- 用户可将候选项接受到规则草稿
- 候选项不会直接覆盖正式生效规则

## 12. 风险与应对

### 12.1 规则模型过松

风险：

- 前端难以统一渲染
- 后端难以稳定校验
- 模板候选结果难以落库

应对：

- 第一阶段先统一字段结构与角色结构
- 明确 `exact / oneOf / range / unset` 四种模式

### 12.2 模板候选可信度不足

风险：

- 用户不信任自动提取结果
- 样本文档中的异常格式污染建议值

应对：

- 不直接生成正式规则
- 必须展示证据、冲突项与置信度
- 候选只进入草稿，由用户确认

### 12.3 一期范围膨胀

风险：

- 规则编辑器尚未成型，就被拉入模板工作台、即时分析、版本中心等复杂范围

应对：

- 一期严格限定为规则配置页 + 合法性校验
- 不引入真实文档重跑闭环
- 不引入复杂版本管理

## 13. 推荐实施顺序

1. 重构规则库内部数据模型，形成 `RuleLibrary / RuleDraft`
2. 落规则配置页
3. 落规则合法性校验
4. 实现 `RuleProposal` 数据结构与接口
5. 实现模板候选提取
6. 实现“接受候选到草稿”交互

## 14. 成功标准

若本方案成功，项目将获得以下能力：

- 支持多模板规则库，而非单一内置规则
- 业务用户可独立维护规则，无需直接操作 JSON
- 模板扩展由“人工写规则”升级为“自动提取候选 + 人工确认”
- parser / validator / web 配置页之间共享一套稳定规则模型

这将直接降低新模板接入成本，并为后续分析工作台和自动化模板扩展打下稳定基础。
