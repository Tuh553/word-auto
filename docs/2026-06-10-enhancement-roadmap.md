# word-auto 优化与功能增强路线建议

> 日期：2026-06-10　范围：基于当前代码库现状的全面盘点
> 与 [`plans/2026-06-09-rule-library-and-template-prd.md`](./plans/2026-06-09-rule-library-and-template-prd.md)
> 互补：PRD 聚焦「规则库可视化」单条主线，本文覆盖**解析引擎 / 校验能力 / 工程质量 / 产品形态**全栈，
> 用「价值 × 成本 × 风险」给出取舍与推荐路线。**仅为规划，未改动代码。**

## 0. 现状速览

| 维度 | 现状 |
| --- | --- |
| 形态 | 合规**检测**（只读、低风险）；纯前端浏览器内解析，文件不上传 |
| 引擎铁律 | 绝不用 Word COM，纯 OOXML（`fflate` 解 zip + `fast-xml-parser` 解 XML） |
| parser | docx→文档模型；样式继承（直接→样式→`basedOn`链→`docDefaults`）；主题字体回填；`sectPr` 页面设置；分节页码；页眉文本 |
| validator | 角色识别状态机（`classify.ts`）；逐段比对（`validate.ts`）；文档级检测；规则合法性校验 `lintRuleLibrary`（新）；规则模型 + 兼容层 |
| web | 四步检测流程 + `docx-preview` 预览高亮；规则配置页骨架（新） |
| 测试/CI | validator 21 例单测（仅 `rules`/`lint`）；CI 仅 `build`+部署，无测试/类型门禁 |

整体判断：**解析与检测的核心链路已跑通且对账过标准模板**，价值已验证。当前最大短板有三类——
①**工程地基薄**（测试金字塔倒置、CI 无门禁、parser 类型不安全）；
②**能力断层**（规则模型支持 4 种取值模式，但检测引擎只认其中一种）；
③**覆盖盲区**（表格、特殊正文元素、章节尾部识别）。下面分层展开。

## 1. 工程质量与可维护性（地基，最高优先）

地基不牢，后续每加一个检测项都在放大回归风险。这一层投入产出比最高。

### 1.1 测试金字塔倒置 —— 检测命脉零覆盖

- **现状**：唯一测试是本轮新增的 `validator` 的 `rules.test.ts` / `lint.test.ts`（21 例，纯数据逻辑）。
  而真正决定**检测准确率**的 `parser`（`ooxml` / `resolve` / `styles` / `theme` / `units`）、
  `validator/validate.ts`、`classify.ts` —— **全部零测试**。
- **风险**：样式继承、单位换算、角色识别任何一处回归都不会被发现，只能靠 `pnpm check` 肉眼看报告。
- **建议**：
  - **单元测试**：`units`（twips/half-point/chars 换算边界）、`resolve`（继承覆盖优先级）、
    `theme`（`a:ea` 为空回退 `Hans`）—— 这些是纯函数，最易测、最该测。
  - **金标准/快照测试**：`templates/source/*.docx` 已是现成样本。对它们跑 `parseDocx` + `validateDoc`，
    把「分类结果 + 问题清单」固化为快照；任何解析/规则改动若改变金标准输出即告警。
  - **classify 用例化**：构造若干段落序列（封面→摘要→正文→参考文献→致谢），断言角色判定，
    把已知缺陷（见 §3.1）变成红灯。
- **成本**：中　**价值**：高（解锁后续所有改动的安全网）

### 1.2 CI 缺质量门禁

- **现状**：`deploy.yml` 只 `pnpm build` + 部署 Pages；`pnpm check` 实为 `tsx` 跑 demo 报告，**不做类型检查**。
- **建议**：新增 CI workflow（PR + push 触发）串联 `tsc --noEmit`（parser/validator/web，本轮已验证可全绿）
  + `pnpm test` + `pnpm build`。把「类型 + 测试」设为合并门禁，部署只在门禁通过后进行。
- **成本**：低　**价值**：高

### 1.3 parser 类型不安全（`any` 泛滥）

- **现状**：`ooxml.ts` 的 `parseXml` 返回 `any`，`wp`/`sect`/`rPr` 等节点操作通篇 `any`。
  本轮修的 `SectionProps` 写入坍缩就是这类隐患的冰山一角。
- **建议**：为 OOXML 关键节点（`w:p` / `w:pPr` / `w:rPr` / `w:sectPr` / `w:tbl`）定义**最小输入类型**
  或引入运行时 schema 校验，把「解析层的结构假设」显式化，避免字段拼写/层级错误静默漏读。
- **成本**：中　**价值**：中（长期可维护性 + 减少隐藏漏报）

### 1.4 CLI 可用性

- **现状**：`apps/cli/src/main.ts` 硬编码 `E:/Claude code/docs` 绝对路径作默认值；无参数校验/帮助/退出码。
- **建议**：去硬编码（改为必填参数或相对 demo 资源）、加 `--help` 与非零退出码（便于脚本/CI 调用）、
  支持目录批量、可选输出格式（json/纯文本）。
- **成本**：低　**价值**：中

### 1.5 错误处理与可观测性

- **现状**：`parseDocx` 对缺 `document.xml` 抛错，但对损坏 zip / 加密 docx / `.doc` 旧格式缺乏友好分流；
  web 端统一 `catch` 成一句「检测失败」。
- **建议**：分类错误（非 docx / 加密 / 损坏 / 不支持的旧格式），给用户可操作提示；
  解析阶段记录「跳过了哪些无法识别的结构」，避免「静默漏检」被误认为「全部合规」。
- **成本**：低　**价值**：中

## 2. 解析引擎 parser 增强

检测能力的上限由 parser 能「看见」多少决定。当前盲区集中在表格、页眉页脚结构、编号与域。

### 2.1 表格内段落（已知最大盲区，建议一次重构解决）

- **现状**：`parseDocx` 只取 `body["w:p"]`（`index.ts`），**表格内 `w:tbl > w:tr > w:tc > w:p` 完全丢失**；
  且段落与 `sectPr` 是分别遍历收集，文档真实顺序信息有限。
- **建议（一石多鸟）**：改用 `fast-xml-parser` 的 `preserveOrder` 模式重构解析层，一次性解决：
  ①表格内段落提取；②`w:p` 与 `w:tbl` 交错的**真实文档顺序**；③分节边界与段落的精确对应。
  表格段落需带「位于表格内」标记，供校验按表格场景降噪（见 §3.2）。
- **风险**：高（触及解析核心，AGENTS 已标注）。**强依赖 §1.1 金标准测试先就位**，否则不可控。
- **成本**：高　**价值**：高

### 2.2 页眉/页脚结构化解析（替代正则）

- **现状**：`index.ts` 用正则 `/<w:t[^>]*>([^<]*)<\/w:t>/g` 抓页眉文本，绕过 XML 解析——
  漏掉 `xml:space` 对象形式、域代码（页码）、左右分区（`left_text` vs 章节名）。
- **建议**：用已有 `parseXml` 正式解析 `header*.xml`/`footer*.xml`，区分左/中/右分区、识别页码域；
  顺带支撑「页脚页码字体/字号」等规则（标准模板要求页码 Times New Roman 小五号居中）。
- **成本**：中　**价值**：中

### 2.3 列表与自动编号（`numbering.xml`）

- **现状**：未解析 `numbering.xml`；标题/列表的自动编号、题序连续性无从校验。
- **建议**：解析编号定义与级别，支撑「标题题序连续」「图表题注连号」（见 §3.4）等高价值检测。
- **成本**：中　**价值**：中

### 2.4 域 / 题注 / 交叉引用 / 脚注尾注

- **现状**：域（`w:fldSimple`/`w:instrText`）、题注、交叉引用、脚注尾注均未建模。
- **建议**：至少识别其存在与文本，支撑目录域、页码域、题注编号检测；也是远期自动套版**必须无损保留**的结构。
- **成本**：中-高　**价值**：中

### 2.5 run 级混排与字符级检测

- **现状**：校验以段落 `effective` 为粒度；段落内**部分 run** 字体/字号不一致（中英混排、个别字加粗）无法定位。
- **建议**：保留 run 级 `props`（已有 `Paragraph.runs`），增加「段内格式一致性」检测，定位到 run/字符区间。
- **成本**：中　**价值**：中

### 2.6 图片 / 公式 / 嵌入对象

- **现状**：未识别 `w:drawing`（图片）、OMML 公式（`m:oMath`）、嵌入对象。
- **建议**：识别其存在与所在段落，避免把图/公式所在段当普通正文误报；为图表题注配套检测打基础。
- **成本**：中　**价值**：中

## 3. 校验引擎 validator 增强

### 3.1 角色识别鲁棒性（线性状态机的尾部问题）

- **现状**：`classify.ts` 是线性状态机，进入 `references` 后，**致谢/附录全被标 `reference_body`**（已知缺陷）；
  封面区识别靠启发式 `COVER_HINT` 正则。
- **建议**：
  - 补章节边界词：`致谢` / `Acknowledg(e)ment` / `附录` / `Appendix` / `攻读学位期间成果` 等，
    各自映射到独立角色或「不校验」。
  - 中期可从「单遍线性」升级为「**先分块再定角色**」：先用强信号（分页符、大纲级别、样式名、关键词）
    切文档块，再在块内定角色，降低顺序耦合带来的连锁误判。
- **成本**：低（补词）→ 中（分块）　**价值**：高

### 3.2 特殊正文元素（消除已知误报）

- **现状**：图表注释/资料来源（9pt）、公式编号被按正文 12pt 校验 → **系统性误报**（README 已知）。
- **建议**：结合 §2.1 表格标记与 §2.6 图/公式识别，为「图表题注」「资料来源」「公式行」建独立角色与规则，
  从源头消除这类误报。
- **成本**：中（依赖 parser 增强）　**价值**：高（直接提升报告可信度）

### 3.3 ⚠️ 规则模型与检测引擎的能力断层（关键一致性缺口）

- **现状**：规则模型与配置页/`lintRuleLibrary` 已支持 **`exact` / `oneOf` / `range` / `unset`** 四种取值模式，
  但检测路径经 `toLegacyRuleLibrary` **只保留 `exact`** —— 即用户在配置页设了「字号 ∈ {10.5, 12}」或
  「行距 18~22pt」，`validate.ts` 也**检测不到**（`oneOf`/`range` 被静默丢弃）。
- **影响**：配置页能配出检测引擎不认识的规则，体验割裂；这是「规则可视化」主线的隐性坑。
- **建议**：让 `validate.ts` **直接消费 `EditableRuleLibrary` 的 `RuleValue`**（按 `mode` 比对），
  弃用「降级到 legacy 单值」的中间层，使配置能力与检测能力对齐。属于结构性改进，建议在配置页可编辑值落地**之前**完成。
- **成本**：中　**价值**：高

### 3.4 文档级一致性检测（高价值新检查项）

当前文档级仅页边距/页眉页脚距/纸张/分节页码/页眉文本。可扩展（规则库 JSON 已含相关字段）：

| 检查项 | 依据/数据来源 | 依赖 |
| --- | --- | --- |
| 目录条目与正文标题一致（文本/页码/层级） | TOC 域 + 实际标题 | §2.4 域解析 |
| 标题题序连续（1→2→3 无跳号） | 标题文本/编号 | §2.3 编号 |
| 图表题注连号 + 与正文引用呼应 | 题注编号 | §2.3/§2.6 |
| 参考文献条数 / 外文占比达标 | JSON 已有 `minimum_count`/`fraction` | 现有能力即可 |
| 关键词数量（3-5 个） | JSON 已有 `recommended_count` | 现有能力即可 |
| 摘要字数区间 | JSON 已有 `recommended_word_count` | 现有能力即可 |
| 孤行/寡行、标题落页底 | 需分页信息（难，docx 不含布局） | 高难，建议缓做 |

- **建议**：先做「条数/字数/占比」这类**仅需现有解析即可落地**的统计型检查（JSON 里规则都已备好，性价比极高），
  再逐步做依赖编号/域的结构型检查。
- **成本**：低（统计型）→ 中　**价值**：高

### 3.5 检测结果增强（可解释性 + 可操作性）

- **规则依据未利用**：规则库 JSON 的 `source.provenance` 保存了**每条规则的原始模板批注原文**
  （如「摘要…黑体，三号，大纲级别1级，居中…」），目前被 `[k: string]: unknown` 吞掉、报告里完全没展示。
  **建议**在每条问题旁展示「规则依据原文」，大幅提升用户信任度——这是几乎零成本的可解释性增益。
- **修复建议**：`Issue` 现有 `expected`/`actual`，可加一句**人话修复指引**（「在『开始』选项卡把字号设为小四」），
  并标注「可一键修复 / 需手动」，为 §5.1 自动修复铺路。
- **置信度/分组**：角色识别为启发式，可给低置信项打标；报告支持按角色/严重级/字段分组与排序。
- **成本**：低-中　**价值**：高

## 4. 规则库与配置产品化（承接 PRD，补全主线）

| 子项 | 阶段 | 说明 | 价值/成本 |
| --- | --- | --- | --- |
| 字段值编辑控件 | PRD 阶段4 | 精确值数值输入、字体下拉、对齐枚举、范围输入，替换当前只读展示（配合 §3.3） | 高 / 中 |
| 规则方式 `mode` 切换 | 阶段4 | exact/oneOf/range/unset 切换 UI | 中 / 中 |
| 草稿/发布态 | 阶段2 | `localStorage` 草稿 + 发布动作；**发布后回灌检测链路**（当前配置不影响检测） | 高 / 中 |
| 多规则库管理 | 阶段5 | 列表/切换/新建/复制；BOM 兼容已就绪 | 中 / 中 |
| 规则库导入/导出 | 新增 | 上传自定义 JSON（去 BOM + `lint` 校验 styles）、导出当前库 | 高 / 低 |
| 模板候选提取 | 阶段6-7 | 上传 .docx → parser+classify 聚合 → 候选值/覆盖率/冲突/置信度 → 接受到草稿 | 高 / 高 |

- **要点**：①§3.3 的引擎对齐应先于「可编辑值」，否则配了不生效；
  ②「导入自定义规则库 + `lint` 校验」性价比极高，是多模板生态的最小可用起点；
  ③模板候选提取是把「人工写规则」升级为「自动提取 + 人工确认」的关键跃迁，但依赖解析与统计稳定。

## 5. Web 应用与交付体验

### 5.1 报告导出（亮点：带批注的 docx）

- **建议**：除 PDF/Excel 清单外，**用纯 OOXML 生成「带 Word 批注的 docx」**——把每个问题作为批注
  （`w:comment`）锚定到对应段落回插。这既是高价值交付物（用户在 Word 里逐条看批注改），
  又**不改写正文**、不违背产品根基，是纯 OOXML 路线的自然延伸，也是迈向 §6 自动修复的安全跳板。
- **成本**：中　**价值**：高

### 5.2 批量检测

- **建议**：一次上传多个 .docx，汇总合规概览 + 单篇下钻；适配「导师/教务批量预审」场景。
- **成本**：中　**价值**：中-高

### 5.3 预览与高亮增强

- **现状**：已攻克 `docx-preview` 行距塌缩/重影（见 PROGRESS）；高亮按段落原文匹配。
- **建议**：默认定位到首个问题；问题列表与预览双向联动滚动；run 级问题高亮到字符区间（配合 §2.5）。
- **成本**：低-中　**价值**：中

### 5.4 性能（大文档不阻塞 UI）

- **现状**：`parseDocx` 同步 `unzipSync` + 全量解析，大文档会阻塞主线程（纯前端尤其敏感）。
- **建议**：解析与校验移入 **Web Worker**，主线程只渲染；大文档分块/惰性解析；展示进度。
- **成本**：中　**价值**：中

### 5.5 引导式修复

- **建议**：基于 §3.5 修复建议，提供「问题 → 在 Word 哪里改」的图文引导；与 §5.1 批注导出形成「看报告→改文档」闭环。
- **成本**：中　**价值**：中

## 6. 平台化与远期（高风险 / 高回报，谨慎推进）

| 方向 | 说明 | 风险 |
| --- | --- | --- |
| 自动套版 / 一键修复 | 从「检测」走向「无损改写」——务必保留分节/域/题注/交叉引用/编号 | **极高**（产品根基级，需 §2.4 完备 + 充分回归） |
| 服务化 API / SaaS | 把 parser+validator 封装为无状态服务（已具备并发前提：无 COM） | 中（隐私：当前不上传是卖点，服务化需明确边界） |
| 规范生态 | 多校/多单位/多语言规范库；社区贡献规则库 | 中（依赖 §4 多规则库 + 候选提取成熟） |
| Word/WPS 插件 | 在编辑器内实时检测（Office.js / WPS 加载项） | 中-高（另一套运行环境） |

- **底线**：自动套版是远期、最高风险阶段，**严禁为实现它而引入 Word COM**；
  必须建立在「域/题注/交叉引用」完整建模 + 金标准回归之上。检测与改写应始终是**两条可分离的产品线**。

## 7. 优先级矩阵与推荐路线

### 7.1 价值 / 成本 / 风险一览

| 项 | 价值 | 成本 | 风险 | 依赖 |
| --- | --- | --- | --- | --- |
| §1.1 核心链路测试（金标准+单元） | 高 | 中 | 低 | — |
| §1.2 CI 质量门禁 | 高 | 低 | 低 | §1.1 |
| §3.3 规则模型↔引擎对齐 | 高 | 中 | 中 | — |
| §3.4 统计型文档检测（字数/条数/占比） | 高 | 低 | 低 | — |

## 8. 守住的原则（贯穿所有迭代）

1. **绝不引入 Word COM**——纯 OOXML 是全部价值所在（跨平台/可并发/可纯前端）。
2. **检测与改写分离**——自动套版高风险，不得为它牺牲检测产品线的稳定。
3. **规则可解释**——每条问题应能回溯到规则依据（provenance），不做「黑箱判定」。
4. **隐私即卖点**——「文件不上传、浏览器内处理」是差异化优势，服务化需谨慎划清边界。
5. **测试先行于重构**——尤其 §2.1 这类触及解析核心的改动，金标准回归不可省。
